# How cPanel Redis Manager Works

A technical walkthrough of the purchase, licensing, installation, and operational model.

---

## 1. Purchase

cPanel Redis Manager is licensed per server at a one-time price of $29.95 USD.

Purchase is completed through the UnderHost client portal:

**[customerpanel.ca/client/store/addons-license-script/redis-manager](https://customerpanel.ca/client/store/addons-license-script/redis-manager)**

After purchase, you receive:
- A license key (either standard `UHREDIS-` or IP-locked `UHREDISIP-` format)
- An activation email with your unique installation instructions

---

## 2. Licensing

cPanel Redis Manager uses a server-bound license system.

**Standard keys (`UHREDIS-`):**
Validated by key only. Can be used on any server.

**IP-locked keys (`UHREDISIP-`):**
Validated against the server's public IP address in addition to the key. If you migrate servers, the IP can be updated through your UnderHost client area.

**Validation mechanism:**
The license key is stored in a protected file on the server (`chmod 600`, owned by root). On each page load, the plugin validates the key against the UnderHost license server. Validation responses are cached locally for one hour. A grace period allows continued operation if the license server is temporarily unreachable.

The license key is never embedded in PHP source code and is never transmitted to end users.

---

## 3. Installation

Installation requires root SSH access to the server.

After purchase, you receive your installation instructions by email. The installer:

1. Verifies your license key against the UnderHost license server
2. Detects the server operating system (AlmaLinux, CentOS, CloudLinux, Ubuntu)
3. Installs the Redis server package via the system package manager if not already present
4. Configures the system Redis service to bind to localhost only
5. Deploys the cPanel plugin to the Jupiter theme directory
6. Registers the plugin with the cPanel plugin system
7. Writes the license key to the protected `.license` file
8. Runs any pending migrations from previous versions
9. Cleans up temporary files

The installer is **idempotent** — it is safe to run multiple times and handles upgrades automatically. If an existing installation is detected, the installer enters upgrade mode, preserves the license key and user data, and applies only the changes needed.

**Installer requirements:**
- Root access
- `curl` and `unzip` available
- cPanel and WHM installed
- Minimum 2 GB RAM (4 GB recommended)

---

## 4. cPanel Integration

After installation, cPanel Redis Manager appears under **Software** in the cPanel interface for every user on the server.

Each cPanel user sees only their own Redis instance. They cannot see or interact with any other account's instance.

**From the cPanel panel, users can:**
- Initialize their Redis instance (first-time setup)
- Start and stop their instance
- View their connection details (host, port, auth key)
- Check real-time stats (memory usage, connections, latency)

**The user never needs to:**
- Edit configuration files manually
- Know which port their instance is on (it is assigned automatically)
- Manage Redis authentication manually

---

## 5. Default Configuration

When a user initializes their Redis instance for the first time, the plugin generates a configuration file automatically.

**Default values:**
```
bind             127.0.0.1          # Localhost only — never exposed to the network
port             <auto-assigned>    # Unique port assigned per account
requirepass      <auto-generated>   # 32-character hex key
maxmemory        1536mb             # Overridable by WHM global config
maxmemory-policy allkeys-lfu        # Optimal for general-purpose caching
databases        5
pidfile          ~/.cpanel/plugin/redis/redis.pid
dir              ~/.cpanel/plugin/redis/data/
```

If a WHM administrator has configured server-wide defaults via the WHM Global Config panel, those values are used in place of the built-in defaults for all new instances.

## 6. Operational Model

### Process Isolation

Each instance runs as the cPanel account user — not as root and not as a shared service user. This ensures that if one account's Redis process crashes, it does not affect any other account.

### Keep-Alive via Cron

A per-user cron job checks the instance every minute and restarts it if it is not running. The cron is state-aware: if the user or an administrator intentionally stops the instance (via the plugin UI), the state file is removed and the cron will not attempt a restart. The instance only restarts automatically if it crashed unexpectedly.

### Status Detection

Instance status is determined by attempting a TCP connection to the configured port. If the port responds, the instance is running. If not, it is not. This is more reliable than PID file checks because PID files can become stale after a crash.

### WHM Administration

Server administrators access a separate WHM panel that provides:
- A view of all cPanel accounts and their Redis status
- Controls to start, stop, or manage any account's instance
- Password rotation per account
- Global configuration management
- Backup controls
- CSF firewall integration
- A web-based Redis CLI for admin use

### Upgrades

Upgrades are applied by re-running the installer. The installer detects the installed version, compares it with the package version, backs up the existing plugin files, deploys the update, preserves the license key and all user data, and runs any migration scripts needed for the new version.

User Redis instances continue running throughout an upgrade. No Redis data or configuration is modified during an upgrade.

---

## 7. Security Model

- All Redis instances bind to `127.0.0.1` — never exposed to the public network
- Each instance has a unique, randomly generated authentication key
- Config files are `chmod 0640` — not world-readable
- The license key file is `chmod 600`, owned by root
- All admin operations in WHM use authoritative root-owned metadata, never trusting user-writable files
- CSRF tokens protect all state-changing actions
- The web-based CLI uses `proc_open()` with an argument array — no shell injection is possible
- An allowlist of safe Redis commands is enforced for the web CLI

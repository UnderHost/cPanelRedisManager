# How cPanel Redis Manager Works

A technical walkthrough of the purchase, licensing, installation, and operational model.

---

## 1. Purchase

cPanel Redis Manager is licensed per server at a one-time price of **$29.95 USD**.

Purchase is completed through the UnderHost client portal:

**[customerpanel.ca/client/store/addons-license-script/redis-manager](https://customerpanel.ca/client/store/addons-license-script/redis-manager)**

After purchase, you receive:

- Your license key
- Access to the client area for license management
- Installation instructions by email

---

## 2. Licensing

cPanel Redis Manager uses a server-bound license model.

### License key formats

- `UHREDIS-` - standard server license
- `UHREDISIP-` - IP-locked server license

### How validation works

- The license key is stored in a protected server-side file
- Validation is performed against the UnderHost licensing system
- Responses are cached locally
- A grace period allows continued operation if the license service is temporarily unreachable

The license key is never embedded in public documentation or exposed to end users.

### Current behavior

For the current release line, validation is designed to be lightweight and reliable for production hosting environments:

- local cache avoids external validation on every page load
- WHM and cPanel contexts share the same validation logic
- background refresh support is part of the 2.2 release line

---

## 3. Installation

Installation requires root SSH access.

After purchase, the installer is provided privately to licensed users. The installation flow:

1. Validates the license key
2. Detects the operating system
3. Installs Redis if required
4. Secures the Redis service for localhost-only operation
5. Deploys the cPanel plugin
6. Registers the plugin with cPanel
7. Configures CloudLinux / CageFS integration where applicable
8. Stores license and metadata safely
9. Runs migrations if needed

The installer is **idempotent**. Re-running it is safe and is also how upgrades are applied.

---

## 4. cPanel User Experience

After installation, cPanel Redis Manager appears in the **Software** section for users on the licensed server.

Each cPanel user only sees and manages their own Redis instance.

### Users can:

- initialize Redis on first use
- start, stop, and restart their instance
- view connection details
- view lightweight status and statistics
- manage safe configuration options as features become available in later release lines

### Users do not need to:

- edit raw Redis configuration manually
- assign ports manually
- manage process supervision directly
- use SSH for day-to-day Redis operations

---

## 5. Instance Provisioning

When a user initializes Redis for the first time, the plugin provisions a per-account instance.

### Typical generated defaults

```text
bind             127.0.0.1
port             <auto-assigned>
requirepass      <auto-generated>
maxmemory        1536mb
maxmemory-policy allkeys-lfu
databases        5
```

If WHM defaults are configured, those values are used for new instances where appropriate.

---

## 6. Operational Model

### Per-account isolation

Each Redis instance runs under the cPanel account system user.

This means:

- separate process per account
- separate config, data, and log paths
- no shared Redis process between customers
- no cross-account access by design

### Lifecycle management

The plugin manages instance lifecycle through the cPanel interface and, in later versions, through WHM as well.

Supported lifecycle operations include:

- start
- stop
- restart
- status inspection

### Keep-alive behavior

A per-user cron job checks whether the instance should be running and restarts it if necessary.

This behavior is state-aware:

- intentionally stopped instances are not restarted
- duplicate restart attempts are prevented with locking
- unexpected crashes can be recovered automatically

### Status and health

The system is moving toward a layered model of runtime state:

- process validation
- TCP connectivity
- authenticated Redis response checks
- health state separate from simple running/stopped state

This is part of the broader stability work in the 2.2 release line.

---

## 7. WHM Administration

The WHM side of the product provides server-wide visibility and control.

Depending on version, this includes:

- server-wide Redis instance overview
- per-account management
- password rotation
- global defaults
- backups
- restricted CLI access
- audit logging and recovery tooling in later releases

WHM actions rely on root-owned metadata rather than trusting user-writable files.

---

## 8. Security Model

The product is designed for multi-tenant hosting environments.

### Core security principles

- All Redis instances bind to `127.0.0.1`
- Each instance uses a unique authentication key
- Sensitive metadata is stored in a root-owned control path
- Privileged actions do not trust user-controlled PID or config files
- State-changing requests require CSRF protection
- Command execution uses safe argument-based invocation, not shell string concatenation

---

## 9. Upgrades

Upgrades are applied by running the installer again.

Upgrade behavior includes:

- installed version detection
- backup of existing plugin files
- preservation of user data and license state
- execution of pending migrations
- idempotent deployment logic

User Redis data is preserved during normal upgrades.
